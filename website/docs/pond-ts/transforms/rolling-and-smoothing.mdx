# Rolling Windows and Smoothing

Sliding-window computations over ordered events. Two primitives:

- **`rolling(...)`** — emit a new series where each event's value is a
  reducer over a window around that event.
- **`smooth(...)`** — numerical trend extraction: EMA, moving average,
  LOESS, box.

Plus two higher-level helpers built on top of `rolling`:

- **`baseline(col, opts)`** — append `avg` / `sd` / `upper` / `lower`
  columns for band charts and outlier detection.
- **`outliers(col, opts)`** — return just the events outside the band.

Coming from pandas? `rolling` ≈ `.rolling(window).agg(mapping)`.
`smooth('ema', ...)` ≈ `.ewm(alpha=...).mean()`.

## Rolling windows

The core idea: for each event in the series, look at a window of
nearby events and compute a reducer. What "nearby" means is the
**window alignment** — trailing, centered, or leading. Pick the one
that matches the question you're answering.

:::note[Diagram candidate]
ASCII first pass; Excalidraw replacement welcome.

```
source events on timeline  ● ● ● ● ● ● ● ● ● ● ● ● ● ● ● ●
                           ─────────────────────────▶
                                       ^ anchor event (i=7)

alignment: 'trailing' (default)
window covers [anchor − duration, anchor]
                           ├───────────┤
                           │   ●─────● │ anchor
                           ├───window──┤

alignment: 'centered'
window covers [anchor − d/2, anchor + d/2]
                                   ├─────┼─────┤
                                   │●─●──●─●──●│
                                   ├─────anchor┤

alignment: 'leading'
window covers [anchor, anchor + duration]
                                         ├───────────┤
                                         │ anchor ● ●│
                                         ├───window──┤
```

Trailing is "what's the average of what I've seen so far"; centered is
"smoothed around now"; leading is "average of what's coming next."
:::

### Two output shapes

Pick the one that matches your output cadence:

```ts
// Per-event rolling: one output event per source event, same key type.
cpu.rolling('5m', { cpu: 'avg' });

// Grid-anchored rolling: one output event per Sequence point.
cpu.rolling(Sequence.every('1m'), '5m', { cpu: 'avg' });
```

The per-event form is the common one. Use the grid form when you want
a fixed output cadence independent of where source events land.

```ts
cpu.rolling('5m', { cpu: 'avg' }, { alignment: 'centered' });
```

Opening events — where the window would extend beyond the series'
ends — emit `undefined` under trailing and centered. Under leading,
trailing events emit `undefined` instead.

### Mapping shapes

Same two forms as [`aggregate`](./alignment-and-aggregation#mapping-shapes):

```ts
// Shorthand — same column in, same column out
cpu.rolling('5m', { cpu: 'avg' });

// Explicit — rename or multi-reducer-per-column
cpu.rolling('5m', {
  avg: { from: 'cpu', using: 'avg' },
  sd: { from: 'cpu', using: 'stdev' },
  peak: { from: 'cpu', using: 'max' },
});
```

The explicit form is how you get a band chart in one pass — `avg` and
`sd` from the same source column without running `rolling` twice. See
the [`baseline()`](#baseline) helper below, which is sugar over this
pattern.

### Custom reducers

Same contract as `aggregate`:

```ts
cpu.rolling('5m', {
  cpu: (values) => {
    const nums = values.filter((v): v is number => typeof v === 'number');
    return nums.length === 0
      ? undefined
      : nums.reduce((a, b) => a + b, 0) / nums.length;
  },
});
```

Mix built-in and custom reducers in the same mapping.

### Rolling complexity

Most built-in reducers are O(1) per event via incremental add/remove.
Median and percentiles are O(N) per event (sorted-array maintenance).
See the [Reducer Reference](./reducer-reference#rolling-window-behavior)
table for the full list.

## Smoothing

Three methods for numeric trend extraction:

- **`'ema'`** — exponential moving average. Lightest weight;
  streaming-friendly. Needs `~1/alpha` samples to converge.
- **`'movingAverage'`** — windowed arithmetic mean. Explicit window
  duration; trailing / leading / centered alignment like `rolling`.
- **`'loess'`** — locally-weighted regression. Slower but better shape
  fidelity for offline smoothing.

:::note[Choosing a smoother]
EMA is the default for dashboards — cheap, reactive, one parameter.
Moving average is the right choice when you want predictable lag
(centered) or strict "average of last N minutes" semantics (trailing).
LOESS is for offline analysis where smoothness matters more than
speed.
:::

### EMA

```ts
const smoothed = cpu.smooth('cpu', 'ema', {
  alpha: 0.3,
  output: 'cpuTrend', // new column; `cpu` passes through unchanged
});
// smoothed.schema gains { cpuTrend, number (optional) }
```

The first ~`1/alpha` rows are noisy — the smoother seeds on the first
raw value and converges over that many samples. Use `warmup: N` to
drop the first `N` output rows once converged (kept rows still computed
against the warmed state — `warmup` trims the output, not the
computation):

```ts
cpu.smooth('cpu', 'ema', { alpha: 0.3, warmup: 4 });
// length === source.length − 4
```

`warmup: 0` is a no-op; `warmup >= series.length` returns an empty
series with the same schema.

### Moving average

```ts
cpu.smooth('cpu', 'movingAverage', {
  window: '5m',
  alignment: 'centered', // 'trailing' | 'leading' | 'centered'
  output: 'cpuAvg',
});
```

### LOESS

```ts
cpu.smooth('cpu', 'loess', {
  span: 0.25, // fraction of series — smaller = more local
  output: 'cpuLoess',
});
```

### Output column

All smoothers take an optional `output` parameter:

- **Omitted** — the target column is replaced in place.
- **String** — a new column is appended; the source column passes
  through unchanged. Useful when you want both the raw and smoothed
  series for a band chart or overlay plot.

## `baseline(col, opts)` \{#baseline\}

Sugar for "rolling avg + sd + upper/lower band in one pass." Appends
four optional number columns to the source schema:

```ts
const base = cpu.baseline('cpu', { window: '1m', sigma: 2 });
// base.schema: [time, cpu, requests, host, avg, sd, upper, lower]

const upperBand = base.toPoints('upper'); // chart-ready { ts, value }[]
const lowerBand = base.toPoints('lower');

const anomalies = base.filter((e) => {
  const v = e.get('cpu');
  const lo = e.get('lower');
  const hi = e.get('upper');
  return v != null && lo != null && hi != null && (v > hi || v < lo);
});
```

### Warm-up and flat windows

- Before the rolling window has enough samples for a meaningful
  baseline, `avg` and `sd` are `undefined`; `upper` and `lower` follow.
- When the rolling window is flat (every sample identical), `sd === 0`.
  `avg` and `sd` still report their real values, but `upper` and
  `lower` collapse to `undefined` — a zero-width band would flag every
  non-equal point as anomalous, which is usually the opposite of what
  you want.

Filters that null-check `upper` / `lower` will do the right thing
through warm-up and flat-window regions.

### Custom column names

If the defaults (`avg`, `sd`, `upper`, `lower`) would collide with
source columns:

```ts
cpu.baseline('cpu', {
  window: '1m',
  sigma: 2,
  names: { avg: 'cpuAvg', sd: 'cpuSd', upper: 'cpuHi', lower: 'cpuLo' },
});
```

Collisions throw at construction, so the rename is mandatory, not
optional.

## `outliers(col, opts)` \{#outliers\}

Just the anomalies. Same schema as the source, filtered to events
outside the baseline's `avg ± sigma * sd` band.

```ts
const anomalies = cpu.outliers('cpu', { window: '1m', sigma: 2 });
// anomalies.length ≤ cpu.length, same schema as cpu

// Bucketed anomaly count per 15s for a bar chart:
const perBucket = anomalies.aggregate(Sequence.every('15s'), {
  cpu: 'count',
});

// Per-host breakdown:
const perHost = anomalies.groupBy('host');
```

Conceptually equivalent to `baseline(col, opts).filter(...)`;
implemented independently (one rolling pass either way). Reach for
`baseline()` directly when you want to render the band too — you get
the `avg` / `upper` / `lower` columns on the same pass.

Events where the rolling window is flat (`sd === 0`) or still warming
up are not flagged — `outliers` and `baseline`'s filter semantics
agree here.

## Chart interop

For drawing, extract a column as `{ ts, value }[]`:

```ts
const pts = cpu.rolling('5m', { cpu: 'avg' }).toPoints('cpu');
// ReadonlyArray<{ ts: number; value: number }>
<LineChart data={pts} />
```

See the [Concepts overview](../concepts/overview#getting-data-in-and-out)
for the full set of data-shape bridges (`toPoints`, `fromPoints`,
`toJSON`).
